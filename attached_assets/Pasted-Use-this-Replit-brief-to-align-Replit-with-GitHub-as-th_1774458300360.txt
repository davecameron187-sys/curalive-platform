## Replit Brief: GitHub-Custodian Migration (CuraLive + Shadow Mode)

### Objective
Reconfigure Replit so GitHub is the single source of truth for all code, config, and release history. Replit should only run/deploy code that originates from GitHub branches.

### Required End State
1. GitHub repo `davecameron187-sys/curalive-platform` is canonical.
2. Replit is connected to the GitHub repo and set to sync from branches, not manual Replit-only edits.
3. Two release tracks are maintained:
   - `main` -> CuraLive production
   - `shadow-mode` -> Shadow Mode environment
4. All updates are made through GitHub PRs and branch merges.
5. Replit deployments only run from approved GitHub branch heads.

### Branching + Promotion Model
1. `develop`: integration/staging branch for active work.
2. `shadow-mode`: Shadow Mode deployment branch.
3. `main`: production deployment branch.

Promotion flow:
1. Feature branches -> PR into `develop`
2. Validated changes -> PR/merge into `shadow-mode`
3. Approved release -> PR/merge into `main`

### Replit Configuration Instructions
1. Connect Replit project to GitHub repository `davecameron187-sys/curalive-platform`.
2. Disable or avoid direct, unsynced Replit edits in production workflow.
3. Create or duplicate Replit deployments:
   - Deployment A (CuraLive): tracks `main`
   - Deployment B (Shadow Mode): tracks `shadow-mode`
4. Set environment variables per deployment (production vs shadow), never hardcoded in repo.
5. Ensure Replit rebuild/redeploy is triggered only after GitHub updates land on the tracked branch.

### Governance Rules
1. No direct push to `main`.
2. PR review required for merges to `main`.
3. CI checks required before merge (build/test/lint as applicable).
4. Hotfixes still go through GitHub branch + PR flow.

### Secrets and Environment Handling
1. Keep secrets in Replit Secrets and/or GitHub Environment Secrets.
2. Maintain separate values for:
   - Production (`main`)
   - Shadow (`shadow-mode`)
3. Commit only `.env.example` with placeholder values; never commit real secrets.

### CI/CD Expectations
1. GitHub Actions runs CI on PRs and pushes.
2. Deploy jobs are branch-specific:
   - `main` -> production deployment
   - `shadow-mode` -> shadow deployment
3. Replit should consume these branch updates cleanly and redeploy.

### Acceptance Criteria
1. A commit merged to `shadow-mode` appears in Shadow deployment without manual file copy.
2. A commit merged to `main` appears in CuraLive production without Replit-only edits.
3. Git history in GitHub fully represents release history.
4. Team can roll back using GitHub commits/tags.

### Rollback Plan
1. Revert problematic merge commit in GitHub.
2. Redeploy from previous known-good commit/tag on `main` or `shadow-mode`.
3. Confirm Replit deployment reflects reverted branch head.

3. Set separate prod/shadow secrets in Replit.
