## Objective

Prove that CuraLive:
- Helps users make better decisions during live events
- Reduces stress, not increases it

---

## Core System Principle (Locked)

CuraLive is an autonomous intelligence gatekeeper.

It decides what is surfaced, filters what is suppressed, and governs what passes through.

It does NOT make decisions, validate decisions, or take actions.

**Audit Truth — Must Always Hold:**
CuraLive detected → User decided

---

## Primary Risk

Noise.

Too many signals creates cognitive overload, stress, distrust, and system abandonment.
Signal discipline is the core design layer of Phase 3.5.

---

## Signal Discipline Rules (Non-Negotiable)

1. If it is not important, it does not exist
2. Less is more — few signals, high action rate
3. Suppression is a feature
4. Late signals are suppressed, not delivered (latency threshold: 1500ms from canonical to dashboard)
5. When in doubt, suppress — never surface

---

## What to Build — Phase 3.5

### Build Immediately

**1. Signal Discipline Core**
- Hard thresholding (confidence + relevance + governance)
- Priority ranking — no chronological stream
- Default suppression bias

**2. Basic Signal Collapsing**
- Merge duplicate and similar signals
- Prevent alert spam during Q&A

**3. Minimum Post-Event Summary**
Using existing data only:
- What was detected
- What was flagged
- What the user did

**4. Outcome Marker (Basic)**
At 24, 48, and 72 hours post-event:
- Prompt: "Did any flagged items require follow-up?"
- Capture response in customer_actions and audit trail
- No inference, no analysis, no conclusions

**5. Presence Indicator (Simple)**
- System is active
- System is monitoring
- Low-visibility, no noise

---

## Confidence Closure Loop

Detect → Interpret → Deliver → Act → Confirm → Reinforce

**Confirm Stage — Strict:**
System may ONLY say:
"No contradiction of governed boundary has been observed based on available signals."

System must NEVER say:
- "You were correct"
- "This was safe"

**Reinforce Stage:**
- Must be specific, not general praise
- Must reference actual decisions made on flagged items

---

## Validate Stage (Mandatory)

Between every event cycle, the system checks whether confidence provided was justified by real-world outcomes.

Uses: Outcome Marker data + existing audit trail.
Does not infer, predict, or generate conclusions.

This is not optional. It is required from the first live event forward.

---

## Known Trust Breakers

- False safe signal later proven risky
- Noise or duplication during live Q&A
- Simultaneous multi-signal overload on a single moment
- Late signals creating confusion
- Silence interpreted as system failure
- No post-event closure
- General reinforcement that feels like praise
- First event that is not controlled
- Borderline signals pushed through by engineers trying to be helpful
